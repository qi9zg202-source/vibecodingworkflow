VibeCoding Workflow

任务驱动、多 Session、可迁移的工作流骨架

这套仓库现在不只是“新业务起手模板”，还包含 task.md、summary handoff、legacy migration、fresh-session driver， 以及一套已迁移完成的 VS Code integration。核心原则仍然不变： startup -> memory -> session，调度权在文件，不在聊天窗口记忆。

最新状态：2026-03-12 已完成 core workflow 验证、fixture 校验、VS Code extension `compile / smoke:session8 / regression:session9 / smoke:session11`，并修复了集成脚本 `passed` 后未及时退出的问题。

标准起手顺序

1

确定项目名、路径、业务目标、技术栈

只给最关键的业务上下文，不要一开始就讲实现细节。

2

明确要求“按 vibecodingworkflow 模板创建”

不要让模型自由发挥，否则容易跳过 PRD、design、memory 或 session 规划。

3

本轮只做 Session 0

只生成流程文档与 prompt 骨架，不写业务代码，不跨到 Session 1。

4

先落 task-centered 控制文件

task.md、PRD.md、design.md、work-plan.md、memory.md、startup-prompt.md、session-0~10-prompt.md、artifacts/session-summary-template.md。

5

后续开发永远从 startup 入口进入

不要直接跑 session-2-prompt.md，而是每次都重新执行 startup-prompt.md。

6

每个 Session 结束先更新 memory，再启动一个新 Session

推荐做法不是在原聊天里自动续跑，而是结束当前会话，在新上下文里重新执行 startup-prompt.md。

常用命令入口

当前仓库已经形成“初始化、迁移、校验、调度、集成验证”五类入口，首页应直接反映这些真实能力。

初始化新项目

./scripts/init-web-vibecoding-project.sh my-project /path/to/parent --git-init

生成新的 task-centered workflow 项目骨架。

迁移旧项目

./scripts/migrate-vibecoding-project.sh /path/to/legacy-project

为旧项目补齐 task.md、artifacts 和 session-spec 输出目录。

验证 workflow

./scripts/verify-vibecoding-workflow.sh

校验模板、driver、references 和生成约束是否一致。

运行 fresh-session driver

python3 ./scripts/run-vibecoding-loop.py /path/to/project --action inspect --json

读取 memory.md 并返回 machine-readable workflow 状态。

Session 0 应该做什么

task.md 定义当前任务边界、业务目标、成功标准和当前轮次真正要交付的对象。
PRD.md 明确业务问题、目标用户、核心目标、验收标准。
design.md 明确模块边界、输入输出、技术方案和验证策略。
work-plan.md 把业务拆成 Session 1 到 Session N，每个 Session 只做一个 Level 3 Task。
memory.md 初始化状态机，设置 next_session、session_gate 和更新规则。
startup-prompt.md 作为唯一入口，负责根据 memory.md 自动路由当前 Session。
session-0~10-prompt.md 提前把边界写清楚，后续开发只需进入对应 Session 执行。
session summary artifacts 每轮结束后把 handoff 证据写入 artifacts/session-N-summary.md，不要只留在窗口记忆里。

后续开发纪律

  • 永远先执行 startup-prompt.md。
  • 不要手动猜“现在该做 Session 3 还是 4”。
  • Session 完成后先更新 memory.md。
  • 然后结束当前会话，启动一个新的 Session / 新上下文。
  • 新会话再次执行 startup-prompt.md。
  • 如果测试失败或阻塞，就留在当前 Session，不得推进。
真正的稳定点不在 prompt 长短，而在于： 调度权在文件，不在窗口记忆。
推荐模式： 一个 Session 完成一个 deliverable，然后重启一个新的 Session。 不把“在原聊天里自动 /clear 后继续跑”作为默认方案。

常见错误

错误 1：直接开始写代码 结果是后面没有可恢复状态机，当前会话结束后很难稳定恢复。
错误 2：不说“按模板创建” 模型可能会漏掉 memory.md 或 session prompt。
错误 3：Session 结束不更新 memory 下一轮就不知道该进哪个 Session。
错误 4：测试没过就推进 流程会失真，后面的 Session 会建立在不稳定前提上。
错误 5：旧项目不迁移，直接假设兼容 当前 contract 已经把 task.md 和 session summary 作为一等资产，建议显式迁移，不做模糊 fallback。

自动调度与集成怎么做

推荐方式不是让同一个聊天窗口在内部自动续跑，而是使用一个 聊天外部的 session driver。 它负责读取 memory.md，判断是否可以推进，然后启动一个新的 Session， 再把 startup-prompt.md 喂给新的上下文。 

Session N 完成
-> 更新 memory.md
-> 外部 driver 读取 memory.md
-> 创建一个新的 Session
-> 在新 Session 执行 startup-prompt.md
-> startup 再决定是否进入 Session N+1
在这个仓库里，对应的原型脚本是： scripts/run-vibecoding-loop.py
如果希望在 IDE 内完成 refresh、prepare、open files 和 runner control， 当前配套实现位于 integrations/vibecoding-vscode-extension/， 架构固定为 Python core + VS Code UI shell。
