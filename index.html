VibeCoding Workflow

任务驱动、多 Session、可迁移的工作流骨架

这套仓库不只是"新业务起手模板"，还包含 task.md、summary handoff、legacy migration、fresh-session driver， 以及一套已迁移完成的 VS Code integration。核心原则： startup → memory → session，调度权在文件，不在聊天窗口记忆。

标准起手顺序

1

确定项目名、路径、业务目标、技术栈

只给最关键的业务上下文，不要一开始就讲实现细节。

2

明确要求"按 vibecodingworkflow 模板创建"

不要让模型自由发挥，否则容易跳过 PRD、design、memory 或 session 规划。

3

本轮只做 Session 0

只生成流程文档与 prompt 骨架，不写业务代码，不跨到 Session 1。

4

先落 task-centered 控制文件

task.md、PRD.md、design.md、work-plan.md、memory.md、startup-prompt.md、session-0~10-prompt.md、artifacts/session-summary-template.md。

5

后续开发永远从 startup 入口进入

不要直接跑 session-2-prompt.md，而是每次都重新执行 startup-prompt.md。

6

每个 Session 结束先更新 memory，再启动新 Session

推荐做法是结束当前会话，在新上下文里重新执行 startup-prompt.md。

测试流程

  • 先验证图可 import，并确认 vibecoding_workflow 可加载。
  • 再验证进程内同步入口 graph.invoke(...)。
  • 再验证进程内异步入口 graph.ainvoke(...)。
  • 之后使用 deterministic mock runner 执行多 Session E2E。
  • 最后通过 LangGraph Local Server 做 HTTP /runs/wait + /state smoke。

执行图节点

  • load_workflow_state：读取 memory.md / task.md 并构造 state。
  • select_session：根据 session_gate 决定 ready / blocked / done。
  • build_runner_input：组装 startup、session prompt 和 runner payload。
  • review_gate：需要人工确认时触发 interrupt()。
  • run_session_task：执行自定义 runner 或默认 claude → codex → dry-run。
  • collect_outputs：检查 summary、manifest 和 post-run memory。
  • persist_workflow_files：写入 vibecoding-loop.jsonl。
  • route_next：重新加载最新 memory.md，决定继续推进或结束。

Session 0 应该做什么

task.md定义当前任务边界、业务目标、成功标准和当前轮次真正要交付的对象。
PRD.md明确业务问题、目标用户、核心目标、验收标准。
design.md明确模块边界、输入输出、技术方案和验证策略。
work-plan.md把业务拆成 Session 1 到 Session N，每个 Session 只做一个 Level 3 Task。
memory.md初始化状态机，设置 next_session、session_gate 和更新规则。
startup-prompt.md作为唯一入口，负责根据 memory.md 自动路由当前 Session。
session-0~10-prompt.md提前把边界写清楚，后续开发只需进入对应 Session 执行。
session summary artifacts每轮结束后把 handoff 证据写入 artifacts/session-N-summary.md，不要只留在窗口记忆里。

后续开发纪律

  • 永远先执行 startup-prompt.md。
  • 不要手动猜"现在该做 Session 3 还是 4"。
  • Session 完成后先更新 memory.md。
  • 然后结束当前会话，启动一个新的 Session / 新上下文。
  • 新会话再次执行 startup-prompt.md。
  • 如果测试失败或阻塞，就留在当前 Session，不得推进。
真正的稳定点不在 prompt 长短，而在于： 调度权在文件，不在窗口记忆。
推荐模式： 一个 Session 完成一个 deliverable，然后重启一个新的 Session。 不把"在原聊天里自动 /clear 后继续跑"作为默认方案。

常见错误

错误 1：直接开始写代码结果是后面没有可恢复状态机，当前会话结束后很难稳定恢复。
错误 2：不说"按模板创建"模型可能会漏掉 memory.md 或 session prompt。
错误 3：Session 结束不更新 memory下一轮就不知道该进哪个 Session。
错误 4：测试没过就推进流程会失真，后面的 Session 会建立在不稳定前提上。
错误 5：旧项目不迁移，直接假设兼容当前 contract 已经把 task.md 和 session summary 作为一等资产，建议显式迁移，不做模糊 fallback。

自动调度与集成怎么做

推荐方式不是让同一个聊天窗口在内部自动续跑，而是使用一个 聊天外部的 session driver。 它负责读取 memory.md，判断是否可以推进，然后启动一个新的 Session， 再把 startup-prompt.md 喂给新的上下文。 

Session N 完成
-> 更新 memory.md
-> 外部 driver 读取 memory.md
-> 创建一个新的 Session
-> 在新 Session 执行 startup-prompt.md
-> startup 再决定是否进入 Session N+1
在这个仓库里，对应的原型脚本是： scripts/run-vibecoding-loop.py
如果希望在 IDE 内完成 refresh、prepare、open files、runner control 和人工复核， 当前配套实现位于 integrations/vibecoding-vscode-extension/， 架构为 Python core + VS Code Dashboard UI。Dashboard 提供 workflow gate / run status 分离显示、runner controls 与 Session 时间线。当前基线不使用 pending_review 持久态；人工复核结果通过 ready / blocked / done 回写到 memory.md。

人工复核与阻塞处理

每个 Session 结束后，仍然要求人工复核，但当前基线不再引入 pending_review 这一额外持久态。Session 完成后，状态直接落到 ready 或 done；如果发现问题，则人工将 session_gate 改为 blocked 并写入 review_notes。 

ready → in_progress → ready
ready → in_progress → blocked → in_progress
ready → done
核心原则：越早的 Session，验收越严格。 Session 0（设计）和 Session 1-3（核心架构）必须严格验收， 因为后续所有 Session 都建立在其基础上。 错误发现越晚，修复成本指数级增长。

▶ ready / done（通过）

Claude 完成 Session，写入 summary + manifest，并将 memory.md 更新为 ready（继续）或 done（全部完成）。调度程序下次读取时按业务 gate 决定是否推进。

✓ 人工复核通过

人工复核通过后，保持 session_gate = ready（或最终的 done）。如果需要推进下一轮，确保 next_session 与 next_session_prompt 已同步到正确值。

⛔ blocked（驳回重做）

如果人工复核发现问题，将 session_gate 改为 blocked 并写入 review_notes。下一轮 Claude 读取 review_notes，针对性修复，重做同一 Session；问题解除后再改回 ready。
